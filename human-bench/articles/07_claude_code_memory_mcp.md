# Claude Codeとの壁打ちが「積み上げ型」に変わった。長期記憶MCPサーバーの開発記

**著者:** テソテソ

**公開日:** 2026/03/25

## きっかけ

Claude Codeを日常的に使用すると、セッションが変わるたびに「前回の議論」や「以前却下した設計案」といった文脈が失われるという課題に直面します。この問題を解決するため、著者は長期記憶MCPサーバー「engram」を開発しました。

## 設計判断：3つの分岐点

### 1. グローバル vs プロジェクト単位

著者は「グローバル1つ + プロジェクトタグでフィルタ」という方式を選択。プロジェクトパスをメタデータとして付与し、検索時に全体とプロジェクト限定を切り替え可能にしました。これにより、プロジェクト横断の知見も蓄積されます。

### 2. Hooks + CLI vs MCPサーバー

MCPサーバー方式を採用した理由は展開の簡単さです。「settings.jsonに数行追加するだけで使える」点が決め手となり、社内展開や他のエディタ（Cursor、Windsurf等）での利用も視野に入れました。

### 3. クリーンアップ戦略

3段階の重複排除を実装：
- **重複排除**：コサイン類似度≥0.90で自動統合
- **容量制限**：全体10,000件超過時に低使用頻度から削除
- **手動prune**：90日未アクセスを明示的に削除

## 技術スタック

```
Claude Code ↔ engram MCPサーバー ↔ SQLite(memory.db)
                        ↓
                  Ruri v3-310m
```

すべてローカル完結で、外部API通信はありません。

## ハイブリッド検索の実装

### キーワード検索（FTS5トリグラム）

3文字単位のスライス索引化により、形態素解析なしで日本語対応。「JWT」「Zustand」といった固有名詞に強い特性を持ちます。

### ベクトル検索（Ruri v3-310m + sqlite-vec）

テキストを768次元ベクトルに変換し、意味的類似度で検索。「認証の方針」で「JWTベースに統一することにした」がヒットするなど、概念的な結びつきを捉えます。

### 統合手法

RRF（Reciprocal Rank Fusion）で両エンジンのスコアを統合し、時間減衰（半減期30日）を適用します。

## 実装上の課題

MCPサーバーの起動時間が約3秒となり、接続タイムアウトに引っかかる問題が発生しました。原因は重いモジュールをモジュールレベルでインポートしていたこと。

**遅延インポート戦略**で解決：重いモジュールは初回のツール呼び出し時にロード。これにより起動時間は0.33秒に改善されました。

## 提供ツール

engramは5つのツールを公開：

| ツール | 用途 |
|--------|------|
| save | メモリ保存 |
| search | ハイブリッド検索 |
| prune | 古いメモリ削除 |
| stats | 統計情報表示 |
| delete | 個別削除 |

## まとめ

engramにより、セッション間での文脈保持が可能になり、AIとの対話が「積み上げ型」に変わります。ローカル完結、ハイブリッド検索、自動クリーンアップ、簡単な展開という特性により、Claude Codeとの壁打ちの質が向上します。
