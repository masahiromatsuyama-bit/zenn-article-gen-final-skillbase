**著者:** geeawa

**公開日:** 2026/03/30

## はじめに

AIエージェントは一度作成すると、翌日には前日の学習内容をすべて失います。イベント駆動で起動するAmbient Agentは、毎回新しいコンテキストで立ち上がるため本質的にステートレスです。

著者が開発しているMoca（Multi-agent Orchestration Chat on AgentCore）というマルチエージェントプラットフォームの運用エージェントが、このステートレス問題に直面しました。3つのWorkerエージェントが毎日AWSのメトリクスを収集してレポートを生成しますが、毎朝同じCLIコマンドを一から組み立て、昨日の異常を覚えていません。

## Ambient Agentの2つのモード

**Statelessモード**：イベントに反応して動くが、実行のたびにコンテキストがリセットされる

**Statefulモード**：実行ごとに経験を記録し、判断基準やツールを自ら更新していく

## ステートレスなエージェントの限界

Moca Ops v1では以下の問題が顕在化しました：

- 毎回ゼロからCLIを組み立てる（トークン浪費、実行時間増）
- 昨日の異常を知らない（継続監視ができない）
- ベースラインがない（固定しきい値のみで誤検知が多い）
- 学習しない（パターン検出の機会損失）

これらはすべて「記憶がない」という一点に帰結します。

## 3層の自己進化メカニズム

### Layer 1: Reusable Scripts（やり方を覚える）

Workerエージェントが初回実行時にAWS CLIコマンドを組み立て、成功したコマンドシーケンスをシェルスクリプトとしてS3に保存します。2回目以降は保存済みスクリプトを実行するだけです。

実行失敗時はスクリプトを削除してゼロから再構築し、環境変化への自己修復を実現します。

### Layer 2: Persistent Instructions（経験を記憶する）

Orchestratorが実行のたびに読み書きする「永続メモリ」です。以下のファイル群をS3で管理：

- `ops-instructions.md`：次回注意すべき点
- `baseline-metrics.json`：通常値の基準
- `known-issues.md`：既知パターンの辞書
- `investigation-log.md`：前回の調査結果
- `changelog.md`：進化ログ

ベースラインは「異常がなかった日のみ更新」というルールで、異常値の混入を防ぎます。更新には指数移動平均（α=0.1）を使用します。

### Layer 3: Self-Evolving Prompt（自分自身を書き換える）

Orchestratorは`manage_agent`というツールを持ち、自身のシステムプロンプトの動的セクションを自律的に更新できます。

プロンプトはIMMUTABLE（不変）セクションとDYNAMIC（自律更新可）セクションに分離：

**IMMUTABLE SECTION**：安全規則、ワークフロー骨格、リソース定義

**DYNAMIC SECTIONS**：ALERT_THRESHOLDS、LEARNED_PATTERNS、STRATEGY_NOTES、EVOLUTION_LOG

安全設計の5原則：
- 不変と可変の分離
- 世代管理（最大5世代）
- 頻度制限（1実行で最大1回の更新）
- 変更ログ（diffサマリーを記録）
- ロールバック可能性

## 運用エージェントのマルチエージェント構成

全体はOrchestrator 1体 + Worker 3体（Compute、Storage、Usage & Cost Collector）で構成されます。

6つのPhaseで実行：

- **Phase 0**：メモリ読み込み
- **Phase 1**：準備（対象日決定）
- **Phase 2**：並列メトリクス収集
- **Phase 3**：異常検知
- **Phase 4**：レポート生成
- **Phase 5**：バグ調査（条件付き）
- **Phase 6**：学習と進化

## 初回実行の結果

初回実行（2026-03-28）では、すべてのレイヤーが動作しました。

Orchestratorはしきい値が現状に合っていないと判断し、自動調整を実行。メモリ初期作成で8つの学習済みパターンを自動記録。

## 4回の実行で確認できたこと

4回目実行で、S3ストレージが45%減少（16.76GB → 9.21GB）。Orchestratorは自力で根本原因を特定：約30日前の大量オブジェクト上書きにより、非現行バージョンがライフサイクルルールで一括期限切れ。データ損失ではなくインフラの想定動作。

エージェントが下した3つの判断：

1. Software Developerへの委譲を却下（アプリケーションバグではないため）
2. ベースラインの「リセット」（EMA追随ではなく一括置換）
3. 新ルール追加（S3サイズの30%以上の日次減少を記録）

## 初回 → 4回目の変化

| 観点 | 初回 | 4回目 |
|-----|------|-------|
| L1: Scripts | 3本新規作成 | 4回連続再利用、自己修復1回 |
| L2: Known Issues | 2件記録 | 6件蓄積（2件解決） |
| L2: Baselines | 延期 | EMA更新3回 + リセット1回 |
| L3: System Prompt | しきい値3箇所調整 | 累計4回更新、パターン8→15件 |

## 運用から得た学び

**良かったこと**：

1. Layer 1のROIが最高：実行時間とトークン消費を劇的に削減
2. Layer 2は「前回の自分からの手紙」として強力
3. Layer 3は「存在すること自体」に価値

**気をつけるべきこと**：

1. 致命的な値はIMMUTABLEに含める
2. ベースラインの「異常なし条件」定義は慎重に
3. プロンプト更新の「変更量」に上限を設ける
4. IMMUTABLE保護はハードガードレールが理想
5. IAM最小権限原則に従って制約する

## まとめ

Ambient Agentは「作る」のではなく「育てる」時代へ：

- L1：再利用可能なプログラムを書く
- L2：経験則をドキュメントとしてチームで共有
- L3：個の人間としてのスキルや戦略が洗練する

この3層の設計パターンは運用エージェントに限らず、イベント駆動で起動するあらゆるAmbient Agentに適用可能です。
