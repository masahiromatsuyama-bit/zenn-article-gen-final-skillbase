# 良い記事の定義（sense.md）

## 共有したくなる記事とは

- 読み終わった後に「チームSlackに貼りたい」「明日誰かに話したい」と思える
- タイトルを見た瞬間に「あ、自分が困っていたやつだ」とクリックしてしまう
- 読んでいる途中で「これ自分でも試せるな」と手を動かしたくなる

## 著者の存在が透けている記事とは

- 「〜という手法があります」ではなく「〜を試したら〇〇で詰まった」という一人称の体験がある
- 失敗・遠回り・気づきのプロセスが書かれており、著者が実際に手を動かした形跡がわかる
- 「正解だけを教える教科書」ではなく「試行錯誤の記録」として読める

## 読者が得るものが明確な記事とは

- 読む前と読んだ後で、読者の「理解」か「行動」が変わる
- 「なるほど」で終わるのではなく、「やってみよう」か「考え方が変わった」で終わる
- 想定読者（例: Claude Code を業務で使い始めたエンジニア）が3000字で得られる価値が1つ明確

## アンチパターン（やってはいけない）

- 冒頭が「この記事では〜について解説します」で始まる（読者は何を得るかを知りたい、何を説明するかではない）
- 全ての節が同じ構成（概要→コード→説明の繰り返し）で単調になっている
- 「〜が重要です」「〜を意識しましょう」という抽象的な締めで終わる
- 著者の試行錯誤が一切なく、ドキュメントの焼き直しになっている
